# Модель угроз: учётные данные PostgreSQL

## Как хранятся и передаются креды

1. Пользователь вводит хост/порт/БД/логин/пароль в форме подключения
   (`widgets/postgres-connection-form`).
2. Конфиг обфусцируется AES-GCM (`shared/lib/utils/crypto.ts`) и хранится
   в Zustand-сторе; **ключ лежит в sessionStorage той же вкладки**.
3. Перед каждым запросом конфиг расшифровывается на клиенте и передаётся
   аргументом Server Action (`computePgMetrics`, `fetchPgTableData`, …) —
   то есть уходит на сервер приложения **в открытом виде** внутри тела
   POST-запроса (защищён TLS канала браузер↔сервер, если приложение за HTTPS).
4. Сервер приложения использует креды только на время запроса и не сохраняет
   (`withPgClient` закрывает соединение в `finally`).

## От чего схема ЗАЩИЩАЕТ

| Угроза | Статус |
|--------|--------|
| Случайный просмотр пароля в DevTools → Application → Storage | ✅ в storage лежит шифртекст |
| Пароль в бэкапах/синхронизации профиля браузера (localStorage) | ✅ только sessionStorage, очищается с вкладкой |
| Долгоживущий секрет на диске | ✅ время жизни = сессия вкладки |
| MITM на соединении сервер→PostgreSQL | ✅ с Phase 2b сертификат проверяется по умолчанию; отключение — только явный opt-out с предупреждением |
| SQL-инъекция через параметры вычислений | ✅ с Phase 2a: Zod-валидация, whitelist таблиц/колонок из information_schema, значения через `$n` |

## От чего схема НЕ ЗАЩИЩАЕТ (осознанно)

| Угроза | Почему принято |
|--------|----------------|
| **XSS / любой JS на странице**: читает и ключ, и шифртекст из sessionStorage, либо перехватывает расшифрованный конфиг | Клиентское приложение без серверного состояния; полная защита требует серверного vault + handle, что меняет архитектуру деплоя. Решение: честная коммуникация + рекомендация read-only учётки |
| Компрометация сервера приложения | Креды проходят через Server Action в открытом виде — сервер по определению доверенный |
| Пользователь с доступом к открытой вкладке | Физический доступ вне модели |

## Обязательные практики

- Подключать PostgreSQL **только под read-only учётной записью**
  (предупреждение показывается в форме подключения).
- Развёртывать приложение только за HTTPS.
- Не отключать проверку сертификата (`sslAllowInvalidCerts`) вне доверенной сети.

## Альтернатива на будущее

Если потребуется реальная защита от XSS: серверное хранилище секретов
(vault / зашифрованная server-side сессия), клиент оперирует только handle'ом
подключения. Требует stateful-деплоя Next.js.
