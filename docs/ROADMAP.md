### Набор задач 2026-06-20 (фронт)

- 🔥 **Автомасштаб чартов — не приближает значения < 1 (радар).**
  Радарная диаграмма (`widgets/group-view/ui/Chart/GroupRadarChart.tsx`,

  (см. фаза 7: `toDisplayScale`/`toPlotScale`).

- 🔥 **Общие правила условного форматирования в области привязки дашборда.**
  Сейчас CF задаётся отдельно на колонке дашборда и на метрике группы.
  Нужно: правила, заданные для колонки дашборда, автоматически применялись
  к привязанной метрике группы в `/groups/[id]` и наоборот — двунаправленно,
  но ТОЛЬКО в пределах привязки дашборда (группы, привязанные к этому
  дашборду). Точка синхронизации — связка `templateId` ↔
  `virtualMetricBindings` (см. `shared/lib/utils/dashboard-columns.ts`).
  Открытый вопрос дизайна: где хранится «источник правды» правил
  (на шаблоне? на колонке? отдельная сущность правил, наследуемая вниз).

- ✅ **Кастомные абстрактные переменные конструктора → в пул (готово).**
  VisualFormulaBuilder: добавленная переменная попадает в пул (customVars),
  становится переиспользуемой кнопкой и доступна в слотах FN(·); засевается

- ⭐ **Производительность обработки файлов (профилирование сделано).**
  **Готово:** замеры по фазам (`duckdb/excel-parser.ts` read/header/toJson/
  normalize + `worker.ts` insert), лог `[Worker] ⏱️ Import profile`; оптимизация
  чтения заголовков (только первая строка вместо всего листа, ~10%).
  **Замер на 1М CSV (16 кол.):** read 54% · header 10% · toJson 12% ·
  normalize 11% · insert 14% (всего ~47с). **Вывод:** для CSV ~77% уходит на
  SheetJS, который для CSV избыточен (DuckDB читает CSV нативно). НО реальные
  большие файлы пользователя — `.xlsx` (там `XLSX.read` неизбежен), а 1М-CSV —
  тестовый датасет из Python. `normalizeValue` уже корректно тянет и `79,15`,
  и `309514.26`, и коды `01:01` (строкой) — авто-детект DuckDB это сломал бы.
  Дальнейшее ускорение CSV завязано на след. задачу (явные параметры импорта).

- 🔥 **Шаг настройки импорта по preview (в работе: фазы 1–2 готовы).**
  Структура — 3 шага: Источник → Импорт(preview+параметры) → Колонки.
  ✅ Фаза 1: `features/setup-dataset/lib/file-preview.ts` (`buildFilePreview`,
  автодетект разделителя, 13 тестов). ✅ Фаза 2: визард рерайт — `SetupStep`
  +'import', `useSetupWizard` держит файл+preview, `FileUploader` отдаёт файл
  наверх (не импортит сразу), новый `ImportConfigStep` (таблица-preview),
  3-шаговый `SetupStepper`. ⬜ Фаза 3: контролы (разделитель/десятичный/типы)
  + проброс в импорт (CSV→нативный `read_csv_auto`). ⬜ Фаза 4: полировка.
  Исходное описание ниже.

  Пользователь хочет до тяжёлого импорта задать по превью: разделитель CSV,
  десятичный разделитель, тип каждой колонки (и защитить коды-ловушки вроде
  `01:01`/`01:01:01`, которые НЕ время). Даёт контроль+корректность И открывает
  быстрый путь: при явных параметрах CSV уходит в нативный `read_csv_auto`
  (исходный буфер, без SheetJS) → ~47с → единицы секунд; `.xlsx` остаётся на
  SheetJS. План (фазы): (1) лёгкий preview — прочитать N первых строк/байт без
  полного импорта (для CSV — срез буфера; для xlsx — `sheetRows`), показать
  таблицу + автоугаданные параметры; (2) UI настройки (разделитель, десятичный,
  per-column тип) — переиспользовать `ColumnManager`; (3) импорт с параметрами:
  CSV→`read_csv_auto(delim=…, decimal_separator=…)` + явные касты, xlsx→
  текущий путь с переданными типами. Точки входа: `widgets/setup-wizard`
  (`UploadStep`/`ColumnSetupStep`), `features/setup-dataset/model/sync-engine`,
  `duckdb/worker.ts` IMPORT_EXCEL, `duckdb/excel-parser.ts`.

- ✅ **Оптимизация вычислений главного дашборда — профиль снят, узких мест нет.**
  **Замер (1М строк, дашборд 4 группы):** total 17–39мс на пересчёт —
  describe 2–5мс · compile 1–4мс · **exec 13–16мс (доминирует)** · build 0–1мс
  (17мс на первом — холодный JIT, дальше ~0). Вывод: движок уже быстрый,
  `exec` (скан+агрегация DuckDB) ужимать почти нечего, `build`-find'ы не стоят
  ничего после прогрева — оптимизировать НЕ нужно. **Сделано:** кэш схемы
  таблицы (`schemaCache` + `invalidateTableCaches`) — DESCRIBE гонялся на
  каждый из нескольких COMPUTE за клик фильтра, теперь раз на таблицу
  (аудит №12); попутно RELOAD_ARROW/IMPORT_EXCEL стали инвалидировать
  prepared-кэш (был пропуск). **Осталось опционально:** аудит №11 (объединить
  KPI-запрос с дашбордным — но это про число запросов, не про их скорость).

- ✅ **Настройки движка DuckDB (память) — готово.**
  В `entities/app-settings`: `duckdbMemoryLimitMB` (null=авто, persist v3).
  UI — `widgets/settings/ui/EngineSettingsSection`. Проброс: app-слой
  (`client-layout`) подписан на `selectEngineConfig` и зовёт
  `duckdbManager.setEngineConfig` → воркер применяет `SET memory_limit`
  (CONFIGURE_ENGINE), переотправляется при перезапуске воркера и
  переприменяется после каждого initDB.
  ⚠️ `threads` убран (v3): wasm-сборка EH скомпилирована без потоков, любой

## 🦖 Мега-босс (в самом конце, после всех правок и оптимизаций)

**Парсинг файлов-агрегатов (не сырых данных).** Сейчас импорт ожидает плоскую
таблицу сырых строк. Нужно научиться принимать уже агрегированные/иерархические
файлы — как это умеет Excel:

- **Сгруппированная колонка-«шапка» над группами** — над столбцами есть
  объединяющая колонка/заголовок верхнего уровня.
- **Данные сгруппированы по уровням** (группировка строк по уровням, как
  excel-outline): строки несут уровень иерархии, а не только листовые значения.
- **Разные уровни — разный набор колонок.** Один уровень может иметь колонку
  (напр. «Площадь»), другой — нет (там везде пусто). Схема не плоская:
  колонка осмысленна только на части уровней.
- **Предпосчитанные/ручные метрики на уровнях.** На уровне может стоять уже
  готовая метрика — вычисленная заранее или вписанная вручную — которая
  НЕ сходится с суммой своих дочерних строк. Такое значение нельзя
  пересчитывать молча: его нужно **сохранить как отдельную переменную**
  и иметь возможность сравнить с вычисляемой (расхождение «введённое vs
  посчитанное» — само по себе аналитическая ценность).

Затрагивает: парсер импорта (`features/setup-dataset`, `duckdb/worker.ts`
IMPORT_*), модель датасета (`entities/dataset/model/types.ts` — уровни/схема
по уровням), иерархию (`entities/hierarchy`), движок вычислений (хранить и
прокидывать «введённое значение» рядом с вычисленным). Большой объём —
делать строго после производительности и оптимизаций выше.

## Лаг отклика фильтра — КОРЕНЬ НАЙДЕН И УСТРАНЁН

**Симптом:** «клик → таблица» 1000–4500мс, при этом сам COMPUTE 14–177мс.
Даже кэш-хиты занимали ~1000мс — значит блокировался главный поток.

**Корень:** `DuckDbEngine.compute` читал ПОЛНЫЙ Arrow-буфер датасета
(сотни МБ — у пользователя 272 МБ) из IndexedDB на КАЖДЫЙ вызов `compute`
(`get('arrow:'+id)`), хотя буфер нужен лишь для редкого восстановления
таблицы. На клик фильтра это N синхронно-блокирующих чтений по 272 МБ
(дашборд + каждый уровень иерархии), которые насыщали поток и тормозили
даже кэш-хиты соседей.

**Фикс:** буфер грузится ЛЕНИВО — `computeDashboard(params, loadArrowBuffer,
signal)` вызывает загрузчик только в `catch` при `isTableMissing`. В норме
272 МБ не читаются вовсе. (engine.ts + manager.ts).

**Профилирование (оставлено):** `[useComputation] ⏱️ <label> → cache/worker
Nms` и `[HierarchyNodes] ⏱️ <col> → … Nms` — для проверки эффекта и будущих
замеров. Вторичные кандидаты (если лаг останется заметным): debounce/приоритет
таблицы vs иерархии; recharts-рендер (`width(-1)`-варнинги). NB: повторный
DESCRIBE в логах = воркер не перезапущен HMR (нужен hard-reload).
