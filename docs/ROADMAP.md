## 📐 Уроки из Perspective (аудит эталона, 2026-06-23)

Изучили [finos/perspective](https://github.com/perspective-dev/perspective) —
зрелую аналитическую платформу нашего класса (пивоты, агрегаты, формулы,
DuckDB-движок, 10+ типов чартов). Ниже — техники, которые стоит перенять, в
привязке к нашему коду. Цель — более широкая поддержка величин, их
представлений и визуализаций.

- ✅ **Two-stage: row-expression ≠ aggregate — у нас УЖЕ реализовано.**
  Проверка кода (2026-06-23) показала: `preprocessAggregateFormula`
  (`shared/lib/computation/lib/aggregate-formula.ts`) переписывает `MAX(a)/SUM(b)`
  в псевдо-переменные `a__MAX`/`b__SUM` + field-зависимости (колонка+агрегат),
  `query-compiler.ts` эмитит их агрегаты в base-CTE, `formula-to-sql` считает по
  псевдо-переменным, `post-process`/`aggregation` переагрегируют «Итого».
  `fieldBindings` — массив; конструктор групп (`use-group-builder`) уже привязывает
  колонку на КАЖДУЮ переменную формулы. То есть это НЕ разрыв (исходный аудит
  ошибался). Документация: [architecture/aggregate-formulas.md](architecture/aggregate-formulas.md).
  Остаток «Фазы 1b» — только UI пути агрегатного импорта (привязать N колонок к
  одному расчётному показателю в `AggregateStructurePanel`), это авторинг, а не
  движок.

- ⭐⭐⭐ **Единый View-config для 1-D и 2-D — реальный оставшийся шов.**
  Данные уже едины (`BreakdownItem.dateLabel` опционален: нет → 1-D, есть → 2-D),
  расходятся РЕНДЕР и НАСТРОЙКИ чартов: 1-D (`GroupChartsPanel`, серии=метрики,
  bar/radar, учитывает `chartStyle`) vs 2-D (`shared/ui/time-breakdown`,
  серии=категории, ось=время, line, хардкод `SERIES_COLORS`, игнорит настройки).
  Оси различны по смыслу — объединяем НЕ рендереры, а **настройки вида**
  (`ChartViewConfig`: тип чарта, стиль метрики, палитра) + общий стиль-субстрат
  (палитра/тултип/формат/пороги). Закрывает гэпы «2-D не уважает стиль» и
  «палитра групп». **📐 Дизайн-план: [architecture/unified-view-config.md](architecture/unified-view-config.md)**
  (фазы 0–4; Фаза 0 — обвязка тестами pivot строго первой).

- ⭐⭐ **Богатая палитра типов чартов (понравилось пользователю).**
  У эталона из коробки: grid, line, bar, area, scatter, **heatmap, treemap,
  sunburst, candlestick** и др. У нас сейчас bar/line/radar. Кандидаты под наши
  данные: **heatmap** (категория × время — естественно ложится на 2-D-пивот),
  **treemap/sunburst** (доли в иерархии — переиспользует `normalizeBy` % и
  дерево `entities/hierarchy`), area (накопление по времени). Подбор типа стоит
  привязать к структуре данных (одна мера во времени → line/area; доли →
  treemap/pie; две меры → scatter). Точки: `widgets/charts-section`,
  `group-view/ui/Chart/*`, `shared/ui/time-breakdown`. Зависит от единого
  View-config и палитры (ниже) — иначе каждый новый чарт тянет свой код-путь.

- ⭐⭐ **Data Model API: формализованный контракт «движок + overlay».**
  У них один интерфейс Server/Client, конкретный движок транслирует
  `group_by/aggregates/filter` в нативный SQL (DuckDB/Clickhouse), UI не знает
  где считается. У нас трансляция «layout → SQL» размазана, а клиентский
  overlay (значения узлов, `entities/aggregate-nodes/lib/overlay`) — пост-хак
  сбоку. Формализовать как часть контракта движка (что обязателен уметь, что
  опционально). Точки: `duckdb/engine.ts`, `query-compiler`.

- ⭐⭐ **Виртуализованный data-grid (windowing).** Их грид рендерит только
  видимые ячейки + ленивая подгрузка → миллионы строк без лагов. У нас
  `DashboardMetricsTable`/`GroupBreakdownTable` рендерят все строки — первый
  кандидат на тормоза при росте. Решение при упоре: `@tanstack/react-virtual`.

- ⭐ **Streaming / delta-updates по primary key.** `update()` принимает только
  дельту, состояние не пересчитывается целиком. Образец для нашего overlay
  введённых значений, если ввод станет частым/тяжёлым.

- ◌ **Date-bucket функции как чек-лист.** Их `bucket()` (день/неделя/месяц/
  квартал/год) — ориентир для размерностей дат в 2-D. Портировать ExprTK не
  нужно, Apache Arrow-транспорт пока оверкилл (всё локально).

- ⭐ **Изучить UX-организацию настроек чартов и таблиц в примерах эталона.**
  Цель — не код, а **раскладка интерфейса**: как `<perspective-viewer>`
  организует настройки (боковая панель колонок drag-and-drop в Group By /
  Split By / Order By / Where, шестерёнка стиля, переключатель типа
  визуализации, выбор агрегата прямо на «таблетке» колонки, инлайн-настройки
  столбца таблицы). Посмотреть их [примеры](https://perspective-dev.github.io/examples.html)
  и block-галерею: где живёт выбор метрик, где — стиль/палитра, как пользователь
  переключает 1-D↔2-D (перетаскиванием поля в Split By), как заданы пер-столбцовые
  форматы. Что подсмотреть для нас: единая панель настроек вида вместо
  разрозненных контролов (`ChartTypeSelector` + селект даты + чекбоксы серий
  сейчас раскиданы по `GroupViewContent`/`time-breakdown`); «таблетки» полей с
  инлайн-выбором агрегата/формата; перетаскивание поля в «по времени» как жест
  перехода в 2-D. Ложится на `ChartViewConfig` (см. дизайн-план) — сначала
  контракт, потом удобная панель над ним. Артефакт: короткий UX-разбор в
  `docs/architecture/unified-view-config.md` (раздел «UX-ориентиры эталона»).

- ✅ **Чего у них НЕТ, а у нас есть:** «% от итога/макс/среднего/мин» —
  Perspective не делает это встроенным агрегатом. Это наш `normalizeBy`
  (`shared/lib/utils/normalize`) — наша добавленная ценность, не отставание.

## Добавлено пользователем

- ✅ **Импорт агрегата: шаблоны-первыми (создавать шаблоны и привязывать
  колонки), а не переименовывать 130+ метрик — готово.**
  `AggregateStructurePanel` переписана: пользователь создаёт N шаблонов прямо в
  панели (логические показатели «Потребность», «Мощность», …) и привязывает к
  ним колонки **поиском** — вводит «Потребность», видит все подходящие поля,
  добавляет по одному или «Добавить все (N)». Колонка принадлежит одному шаблону
  (привязанная исчезает из пула других). Внутреннее имя колонки остаётся
  уникальным (`группа · колонка`), отображается имя шаблона; непривязанные
  колонки падают на свой шаблон по имени колонки. Сериализация в импорт —
  прежняя `metricTemplateNames` (fullName → имя шаблона), `sync-engine`
  не менялся (дедуп по имени уже был).

- ✅ **Импорт агрегата: выбор агрегации на показатель (SUM/AVG/MIN/MAX) — готово
  (одновходовые).** В `AggregateStructurePanel` у каждого черновика-шаблона —
  блок «Формула и формат»: быстрый выбор агрегации (SUM/AVG/MIN/MAX/COUNT/MEDIAN
  пишет `FN(value)`) + ручная формула. Многополевые отношения (`SUM(a)/SUM(b)`) —
  см. пункт ниже.

- ⭐ **Единый импорт, Фаза 1: формула+формат шаблона при импорте — готово
  (одновходовые).** Черновик-шаблон несёт `formula` (дефолт `SUM(value)`),
  `displayFormat`/`decimalPlaces`/`unit`; одновходовость проверяется через
  `extractVariables` (ровно один алиас), к нему привязывается колонка.
  Сериализация — `AggregateLayoutConfig.metricTemplateSpecs[]` (shared-тип +
  zod + round-trip тест); `createAggregateGroups` берёт формулу/формат/алиас из
  спеки вместо хардкода `SUM(value)`, переиспуёт шаблон по имени+формуле.
  **Осталось (Фаза 1b):** многополевые формулы `SUM(a)/SUM(b)` — отдельный режим
  «расчётный показатель» с привязкой по алиасам (a→колонка, b→колонка) → одна
  метрика; ломает модель «1 колонка = 1 метрика», поэтому вынесено отдельно.

- ✅ **Кросс-столбцовые трансформации (% от итога/макс/среднего/мин) — готово.**
  Пост-обработка результата: значение каждой строки делится на ориентир по
  столбцу ТЕКУЩЕГО вида (в группе — дети уровня, на дашборде — строки-группы),
  показ процентом делает существующий `displayFormat` (доля → %). Строка «Итого»
  остаётся абсолютной (она же ориентир); чарты наследуют нормализацию; значения
  узлов (●) входят в знаменатель как есть. **Хранение — на шаблоне**
  (`MetricTemplate.normalizeBy`, рядом с форматом/CF; единый источник, наследуется
  группой и дашбордом, проходит экспорт/импорт). Чистый хелпер
  `entities/metric/lib/normalize` (`columnReference`/`normalizeColumn`/
  `normalizeVmRows`, 15 тестов); вычисление — пост-пасс на рендере, нигде не
  хранится. Задаётся в `TemplateForm` и в панели агрегатного импорта (выбор
  «% от …» сразу ставит формат «процент»). **Курируемый набор** — расширяемо:
  индекс-к-100 (число), ранг/перцентиль, % от родителя в иерархии — на потом.
  Кросс-строчные нормализации НЕ выразимы формулой метрики (она — скаляр на
  группу), поэтому это отдельный слой, а не формула. **2-D (категория × время)**
  — нормализация ПО ПЕРИОДАМ (доля от столбца каждой даты; обобщает 1-D),
  ячейки/чарт в %, Σ-столбец = доля строки в общем итоге; хелпер переехал в
  `shared/lib/utils/normalize` (нужен и `shared/ui/time-breakdown`).

- ⭐ **Условное форматирование строк (показывать строку или нет).**
  Как окрашивание (правила на шаблоне), но постобработка вычисленных строк:
  по правилу скрывать строку в таблице/чартах (напр. значение == 0, или
  пусто). Живёт рядом с `colorConfig`/CF-правилами шаблона; применяется к
  `breakdown` перед рендером (фильтр в `GroupViewContent`/`GroupBreakdownTable`).
  Должно работать и на введённые значения узлов (режим «считать введённые»).

- ✅ **Замена файла для агрегата (по тем же настройкам разметки) — готово.**
  `AggregateLayoutConfig` теперь хранится на датасете (`DatasetEntry.aggregateConfig`,
  персист в IndexedDB). `replaceDatasetFile` получил ветку агрегата: новый файл
  сплющивается по сохранённой разметке (общий хелпер `buildAggregateImport`),
  узлы пересобираются (`setNodes`), типы колонок навязываются из разметки;
  иерархия и группы сохранены (ID и имена колонок не меняются). Настройки чтения
  проходят через экспорт/импорт конфигов: добавлены в `DatasetConfigExportSchema`,
  `ConfigExport/ImportContext`, применяются к целевому датасету при импорте
  (round-trip покрыт тестом). Замена доступна через существующий UI без отдельной
  кнопки для агрегата.

- ✅ **Введённые значения узлов на дашборде (как в группе) — готово.**
  Overlay-хелперы вынесены в `entities/aggregate-nodes/lib/overlay`
  (`mergeEnteredVms`, `enteredVmValues`) и переиспользуются группой и дашбордом.
  В `DashboardViewContent` каждая строка-группа = узел на текущем пути
  иерархических фильтров: где у узла есть введённое значение, оно перекрывает
  сводку группы (таблица/чарты/статы/форматирование/окрашивание). Тумблер
  «Считать введённые значения» на дашборде (виден для агрегатов с узлами).
  Ограничение: overlay на уровне сводок групп (не на breakdown/динамике по
  времени) — на верхнем уровне без фильтров «Итого»-узел не подставляется
  (как и в группе).

- ⭐ **Единый импорт: один настраиваемый путь для сырых данных И агрегатов.**
  Идея: импорт-панель агрегата фактически умеет и сырые данные (сырые = частный
  случай без каскада ключей). Если добавить туда **создание групп прямо при
  загрузке** и для сырых, можно оставить один способ импорта: пользователь сразу
  настраивает структуру и показатели, дальше донастраивает нашим функционалом.
  Предпосылки (без них не закрыть):
  - ✅ **редактируемые шаблоны прямо в импорте** (формат/знаки/единица) — готово
    в Фазе 1 (блок «Формула и формат» на черновике-шаблоне);
  - ✅ формула шаблона при импорте (одновходовая) — готово (Фаза 1); многополевая
    `SUM(a)/SUM(b)` — **готово (Фаза 1b)**: `CalculatedTemplateSpec` (формула над
    именами колонок + операнды alias→имя), `planAggregateGroups` раскрывает по
    группам с всеми операндами, UI «Расчётные показатели» в `AggregateStructurePanel`
    (формула + привязка алиасов к именам колонок + формат/нормализация);
  - ✅ **создание групп при загрузке для сырых данных** — готово (Фаза 3),
    **ДО импорта** (как агрегат, единый момент): `RawGroupsPanel` управляемая,
    в `ImportConfigStep` на числовых колонках превью, эмитит `RawGroupsConfig`
    (распределение + формула/формат); применяется в `syncFromFile` общим
    `createAggregateGroups` с синтетическими колонками. Пост-импортный шаг убран
    (он сбивал с толку: показывался и для агрегатов с уже созданными группами).
  - ✅ **слияние панелей — прагматично (Фаза 4)**: тоггл «иерархические данные»
    оставлен НЕСУЩИМ (иерархию нужно сплющивать ДО импорта — иначе риск для
    конвейера), но панели сведены к общим примитивам: `TemplateFormatFields`
    переиспользуется и агрегатной, и сырой панелью (убран дубль ~90 строк),
    опыт создания групп единый. Полное слияние лайфцикла сочтено невыгодным.
  Итог: единый опыт импорта; группы создаются при загрузке в обоих режимах.
  **📐 Дизайн-план: [architecture/unified-import.md](architecture/unified-import.md)**
  (фазы 0–4; ключевой инсайт: агрегатный путь — надмножество сырого, общее ядро
  `createAggregateGroups`; Фаза 0 — обвязка тестами строго первой).

## 🦖 Мега-босс (в самом конце, после всех правок и оптимизаций)

**Парсинг файлов-агрегатов (не сырых данных).** Сейчас импорт ожидает плоскую
таблицу сырых строк. Нужно научиться принимать уже агрегированные/иерархические
файлы — как это умеет Excel:

- **Сгруппированная колонка-«шапка» над группами** — над столбцами есть
  объединяющая колонка/заголовок верхнего уровня.
- **Данные сгруппированы по уровням** (группировка строк по уровням, как
  excel-outline): строки несут уровень иерархии, а не только листовые значения.
- **Разные уровни — разный набор колонок.** Один уровень может иметь колонку
  (напр. «Площадь»), другой — нет (там везде пусто). Схема не плоская:
  колонка осмысленна только на части уровней.
- **Предпосчитанные/ручные метрики на уровнях.** На уровне может стоять уже
  готовая метрика — вычисленная заранее или вписанная вручную — которая
  НЕ сходится с суммой своих дочерних строк. Такое значение нельзя
  пересчитывать молча: его нужно **сохранить как отдельную переменную**
  и иметь возможность сравнить с вычисляемой (расхождение «введённое vs
  посчитанное» — само по себе аналитическая ценность).

Затрагивает: парсер импорта (`features/setup-dataset`, `duckdb/worker.ts`
IMPORT_*), модель датасета (`entities/dataset/model/types.ts` — уровни/схема
по уровням), иерархию (`entities/hierarchy`), движок вычислений (хранить и
прокидывать «введённое значение» рядом с вычисленным). Большой объём —
делать строго после производительности и оптимизаций выше.

**📐 План реализации (фазы 0–3): [architecture/aggregate-files.md](architecture/aggregate-files.md)**
— детект структуры → сплющивание листьев (MVP) → таблица узлов с «введённое vs
вычисленное» → разреженная схема. Рекомендация: фаза 0 → 1 → 2 → 3.

## Лаг отклика фильтра — КОРЕНЬ НАЙДЕН И УСТРАНЁН

**Симптом:** «клик → таблица» 1000–4500мс, при этом сам COMPUTE 14–177мс.
Даже кэш-хиты занимали ~1000мс — значит блокировался главный поток.

**Корень:** `DuckDbEngine.compute` читал ПОЛНЫЙ Arrow-буфер датасета
(сотни МБ — у пользователя 272 МБ) из IndexedDB на КАЖДЫЙ вызов `compute`
(`get('arrow:'+id)`), хотя буфер нужен лишь для редкого восстановления
таблицы. На клик фильтра это N синхронно-блокирующих чтений по 272 МБ
(дашборд + каждый уровень иерархии), которые насыщали поток и тормозили
даже кэш-хиты соседей.

**Фикс:** буфер грузится ЛЕНИВО — `computeDashboard(params, loadArrowBuffer,
signal)` вызывает загрузчик только в `catch` при `isTableMissing`. В норме
272 МБ не читаются вовсе. (engine.ts + manager.ts).

**Профилирование (оставлено):** `[useComputation] ⏱️ <label> → cache/worker
Nms` и `[HierarchyNodes] ⏱️ <col> → … Nms` — для проверки эффекта и будущих
замеров. Вторичные кандидаты (если лаг останется заметным): debounce/приоритет
таблицы vs иерархии; recharts-рендер (`width(-1)`-варнинги). NB: повторный
DESCRIBE в логах = воркер не перезапущен HMR (нужен hard-reload).
