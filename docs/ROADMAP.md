Зафиксировано 2026-06-12 по итогам рефакторинга и аудитов.
Приоритеты: 🔥 рекомендовано на ближайшую итерацию · ⭐ высокая ценность · ◌ фоном.

## Добавлено пользователем

### Набор задач 2026-06-20 (фронт)

- 🔥 **Автомасштаб чартов — не приближает значения < 1 (радар).**
  Радарная диаграмма (`widgets/group-view/ui/Chart/GroupRadarChart.tsx`,
  `widgets/charts-section/ui/RadarChartView.tsx`) и, вероятно, бары при
  малых значениях/долях не подгоняют ось — данные «жмутся» у центра.
  Нужен авто-домен оси по фактическому диапазону серий (а не от 0 или
  фиксированного максимума), с учётом процентного масштаба отображения.
  Связано с переводом величин в проценты и масштабом порогов
  (см. фаза 7: `toDisplayScale`/`toPlotScale`).

- 🔥 **Общие правила условного форматирования в области привязки дашборда.**
  Сейчас CF задаётся отдельно на колонке дашборда и на метрике группы.
  Нужно: правила, заданные для колонки дашборда, автоматически применялись
  к привязанной метрике группы в `/groups/[id]` и наоборот — двунаправленно,
  но ТОЛЬКО в пределах привязки дашборда (группы, привязанные к этому
  дашборду). Точка синхронизации — связка `templateId` ↔
  `virtualMetricBindings` (см. `shared/lib/utils/dashboard-columns.ts`).
  Открытый вопрос дизайна: где хранится «источник правды» правил
  (на шаблоне? на колонке? отдельная сущность правил, наследуемая вниз).

- ✅ **Кастомные абстрактные переменные конструктора → в пул (готово).**
  VisualFormulaBuilder: добавленная переменная попадает в пул (customVars),
  становится переиспользуемой кнопкой и доступна в слотах FN(·); засевается
  из формулы при открытии, можно убрать из пула. См. также «(после Фазы B)».

- ⭐ **Производительность обработки файлов (профилирование добавлено).**
  Импорт файла на ~1 млн строк может занимать заметное время. **Шаг 1 (готово):**
  замеры по фазам в `duckdb/excel-parser.ts` (read / toJson / normalize) и в
  `duckdb/worker.ts` IMPORT_EXCEL (insert) — лог `[Worker] ⏱️ Import profile`
  через `logger.info` (dev) с throughput и %-долями. **Шаг 2 (после замеров на
  реальном файле):** оптимизировать самую тяжёлую фазу. Гипотезы: SheetJS
  `XLSX.read`/`sheet_to_json` (аллокация объектов) — основной кандидат;
  `normalizeValue` гоняет несколько регэкспов на ячейку; батч-вставка делает
  лишний round-trip через temp-таблицу + `INSERT INTO SELECT` (можно
  append'ить напрямую, если совпадает схема Arrow между батчами).

- ⭐ **Оптимизация вычислений главного дашборда.**
  Посмотреть, можно ли ускорить элементы дашборда: фильтр и калькуляцию
  (`use-dashboard-computation`, query-compiler, пересчёт при смене фильтра).
  Пересекается с хвостом аудита №11 (KPI в основной запрос) и №15
  (линейные find в компиляторе).

- ✅ **Настройки движка DuckDB (память) — готово.**
  В `entities/app-settings`: `duckdbMemoryLimitMB` (null=авто, persist v3).
  UI — `widgets/settings/ui/EngineSettingsSection`. Проброс: app-слой
  (`client-layout`) подписан на `selectEngineConfig` и зовёт
  `duckdbManager.setEngineConfig` → воркер применяет `SET memory_limit`
  (CONFIGURE_ENGINE), переотправляется при перезапуске воркера и
  переприменяется после каждого initDB.
  ⚠️ `threads` убран (v3): wasm-сборка EH скомпилирована без потоков, любой
  `SET/RESET threads` бросает «compiled without threads». Spill на диск/
  `temp_directory` (OPFS) — отдельная задача при необходимости.

- 🔥 **Sidebar ререндерится «каждый кадр».**
  В отладке видно постоянное обновление `SidebarComponent`
  (`widgets/sidebar/ui/Sidebar.tsx`). Проверено: `duckdbManager.setStatus`
  уведомляет только при смене статуса (`if (this._status === status) return`),
  и `useEngineStatus` шлёт примитив — значит источник НЕ здесь. Смотреть выше
  по дереву: (1) родительский layout/shell, который перерисовывается
  постоянно (подписка на тикающий стор/контекст), при том что
  `SidebarComponent` не обёрнут в `React.memo`; (2) нестабильный селектор
  zustand в layout, возвращающий новый объект (тот же класс бага, что
  чинили в KPI — см. фаза 7). План: профилировать родителя (React DevTools
  Profiler «why did this render»), при необходимости — `React.memo` + аудит
  селекторов в шелле.

- ⭐ **Облагородить 2D-визуализацию при мелких срезах.**
  Когда группировка идёт по слишком мелким срезам (много категорий ×
  размерность дат), чарт выглядит «страшненько» и разрастается за пределы
  экрана. Начать с ограничения самого чарта: фикс. макс-размер контейнера
  + внутренний скролл внутри чарта (а не скролл всей страницы). Затем —
  читаемость (поворот/прореживание подписей, мин. ширина категории).
  Точки входа: `widgets/group-view/ui/Chart/*` (GroupBarChart/GroupRadarChart),
  `widgets/charts-section`, обёртки ResponsiveContainer (recharts).

- ⭐ **Всплывающие окна не адаптированы под тему.**
  Часть поповеров/тултипов/диалогов не реагирует на текущую тему (нет
  `dark:`-вариантов или захардкожены светлые цвета). Нужен аудит
  `shared/ui` (popover/tooltip/dialog) и кастомных тултипов recharts,
  привести к токенам темы. Эталон — `DatasetSwitcher` (там `dark:` есть).

- 🔥 **Возврат со страницы группы — на предыдущую в истории.**
  Сейчас «назад» с `/groups/[id]` ведёт на список групп, но на страницу
  можно прийти и из дашборда. Использовать `router.back()` (история), с
  фоллбэком на `/groups`, если истории нет (прямой заход). Точка входа:
  кнопка «назад» в `widgets/group-view` (GroupPageHeader / страница группы).
  Самый лёгкий фикс из набора.

## 🦖 Мега-босс (в самом конце, после всех правок и оптимизаций)

**Парсинг файлов-агрегатов (не сырых данных).** Сейчас импорт ожидает плоскую
таблицу сырых строк. Нужно научиться принимать уже агрегированные/иерархические
файлы — как это умеет Excel:

- **Сгруппированная колонка-«шапка» над группами** — над столбцами есть
  объединяющая колонка/заголовок верхнего уровня.
- **Данные сгруппированы по уровням** (группировка строк по уровням, как
  excel-outline): строки несут уровень иерархии, а не только листовые значения.
- **Разные уровни — разный набор колонок.** Один уровень может иметь колонку
  (напр. «Площадь»), другой — нет (там везде пусто). Схема не плоская:
  колонка осмысленна только на части уровней.
- **Предпосчитанные/ручные метрики на уровнях.** На уровне может стоять уже
  готовая метрика — вычисленная заранее или вписанная вручную — которая
  НЕ сходится с суммой своих дочерних строк. Такое значение нельзя
  пересчитывать молча: его нужно **сохранить как отдельную переменную**
  и иметь возможность сравнить с вычисляемой (расхождение «введённое vs
  посчитанное» — само по себе аналитическая ценность).

Затрагивает: парсер импорта (`features/setup-dataset`, `duckdb/worker.ts`
IMPORT_*), модель датасета (`entities/dataset/model/types.ts` — уровни/схема
по уровням), иерархию (`entities/hierarchy`), движок вычислений (хранить и
прокидывать «введённое значение» рядом с вычисленным). Большой объём —
делать строго после производительности и оптимизаций выше.

## Быстрые победы (код наполовину готов)

- 🔥 **Экспорт в Excel** — breakdown-таблицы, pivot «иерархия × время»,
  таблица дашборда. Библиотека `xlsx` уже в зависимостях (импорт),
  нужно обратное направление.
- 🔥 **Период-к-периоду** — поля `previousValue` / `change` / `changePercent`
  в `ComputedMetricValue` (shared/lib/types/computation.ts) заведены,
  но никем не заполняются. С группировкой по датам сравнение
  «текущий месяц vs прошлый» на KPI-карточках — естественный шаг.
- ⭐ **Фильтр по диапазону дат** — `ComputationContext.dateRange` существует
  и не используется. Date-picker над дашбордом + WHERE в query-compiler.
- ⭐ **Σ в pivot для AVG/MAX/MIN** — протащить тип агрегации в метаданные
  виртуальной метрики; для AVG показывать среднее, для MAX/MIN экстремум,
  а не сумму (текущая Σ в TimeBreakdownSection осмысленна только для
  SUM/COUNT).

## Среднесрочные фичи

- ⭐ **Сохранение состояния просмотра** — выбранные метрики, размерность дат,
  состав серий чарта сбрасываются при уходе со страницы. Persist
  per dashboard/group.
- ⭐ **Публичные read-only дашборды** — поле `Dashboard.isPublic` есть в типе
  и не работает. Режим «поделиться ссылкой» без редактирования.
- ◌ **Выбор дата-колонки** — сейчас берётся первая с классификацией «Дата»;
  нужен селектор при нескольких.
- ◌ **Алёрты по порогам** — правила окрашивания уже описывают «плохо/хорошо»;
