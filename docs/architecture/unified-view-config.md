# Единый View-config: сведение 1-D и 2-D визуализаций

Дизайн-документ. Статус: **Фазы 0–3 реализованы** (2026-06-27).
- Фаза 1 (общий субстрат: палитра `chart-palette.ts`, `ChartTooltip`,
  `effectiveChartFormat`, пороги) — в коде, читается обоими путями.
- Фаза 3 (палитры: `IndicatorGroup` + `PalettePicker` + `CHART_PALETTES`) — в коде.
- Фаза 2 (единый `ChartViewConfig` на группе: `shared/lib/types/chart.ts`
  `resolveChartView`, поле `IndicatorGroupSchema.chartView`) — реализована:
  типы чартов 1-D и `seriesLimit` 2-D персистятся, `paletteId` переехал в
  `chartView` (legacy-fallback сохранён). Линия/столбцы 2-D — НЕ отдельный
  тоггл, а `chartStyle.kind` выбранной метрики (единый контрол на KPI-карточке,
  как в 1-D).
- Фаза 4 (новые типы чартов — heatmap/treemap) — north star, не начата.

Дата создания: 2026-06-23. Контекст: аудит [finos/perspective](https://github.com/perspective-dev/perspective)
выявил, что у эталона вид (`View`) — один декларативный объект, а 1-D/2-D —
лишь разные комбинации осей одного движка. У нас движок данных уже единый,
а рендер и настройки чартов — два разных кода-пути. Цель документа — описать,
**что именно сводить, что оставить раздельным и почему**, и дать план миграции.

---

## 1. Что есть сейчас (факты из кода)

### Данные — УЖЕ едины
Один хук `useGroupBreakdown` (и аналог на дашборде) выдаёт массив
`BreakdownItem[]`, где у каждого элемента есть опциональный `dateLabel`:

- `dateLabel === undefined` → строка 1-D (категория без времени);
- `dateLabel` задан → ячейка 2-D (категория × период).

Флаг `isTwoDimensional` лишь переключает, какой набор строк показывать.
То есть **разделения на уровне вычислений нет** — это одна таблица фактов.
См. [GroupViewContent.tsx](../../src/widgets/group-view/ui/GroupViewContent.tsx):
`oneDimBreakdown` (фильтр `dateLabel === undefined`) vs `breakdown` (всё).

### Рендер и настройки — РАСХОДЯТСЯ
| | 1-D | 2-D |
|---|---|---|
| Компонент | `GroupChartsPanel` → `GroupBarChart`/`GroupRadarChart` | `TimeBreakdownSection` (`shared/ui/time-breakdown`) |
| Ось X | категории (`item.label`) | время (`dateLabel`) |
| Серии | **метрики** (несколько сразу) | **категории** (top-N), одна метрика |
| Тип чарта | bar / radar (выбор `chartTypes`) | line (хардкод) |
| Стиль метрики | `chartStyle` (столбец/линия) учитывается | игнорируется |
| Палитра | своя в компонентах чартов | хардкод `SERIES_COLORS` в `ui.tsx` |
| Формат/масштаб | `toDisplayScale` + `formatById` | `toDisplayScale` + `effectiveFormat` (дублируется) |
| Пороги | `groupThresholdsByValue` | `groupThresholdsByValue` (дублируется) |
| Тултип темы | в `GroupBarChart` | свой кастомный (чинили отдельно) |

**Вывод:** различие осей (метрики-как-серии vs категории-как-серии) —
**сущностное**, а не косметическое. 1-D отвечает «сравнить категории по
нескольким метрикам», 2-D — «динамика одной метрики по категориям во времени».
Свести их в один компонент-рендерер нельзя без потери смысла.

То, что РЕАЛЬНО задублировано и должно быть общим: палитра, формат/масштаб,
группировка порогов, тема тултипа, и сами **настройки** (тип чарта, стиль
метрики, палитра) — которые сейчас текут только в 1-D.

## 2. Что сводим, что оставляем

**Сводим (общий субстрат, `shared`):**
1. **Палитра серий** — единый источник цветов вместо хардкода `SERIES_COLORS`
   в `time-breakdown` и разрозненных цветов в `Chart/*`. Цвет назначается
   по ключу серии (метрика или категория) детерминированно.
2. **Тема тултипа** — один themed-компонент тултипа на `dark:`-классах
   (сейчас две копии: 1-D и 2-D-фикс).
3. **Формат/масштаб значения** (`toDisplayScale` + выбор `effectiveFormat`
   с учётом нормализации) — общий хелпер, чтобы percent/percent_raw/нормализация
   считались одинаково в обоих путях.
4. **Группировка порогов** — уже общий `groupThresholdsByValue`, но рендер
   ReferenceLine/Label дублируется — вынести общий under-chart примитив.

**Оставляем раздельным (оправданно):**
- Сам выбор осей и компонент чарта: 1-D (серии=метрики) и 2-D (серии=категории,
  ось=время) — разные виджеты. Их не объединяем.

**Связующее звено — единый объект настроек вида (`ChartViewConfig`).**
Вместо разрозненных пропсов (`chartTypes`, `chartStyleByMetricId`, нет палитры)
ввести один сериализуемый конфиг, который читают ОБА пути:

```ts
interface ChartViewConfig {
  chartTypes: ChartType[];                 // bar | radar | line | (area | heatmap…)
  styleByMetricId: Record<string, MetricChartStyle>;
  paletteId?: string;                      // ← закрывает ROADMAP-палитру
  seriesLimit?: number;                    // top-N для 2-D
}
```

Это и есть «View-config» в нашем масштабе: не переписываем движок, а делаем
**настройки вида единым контрактом**, который одинаково понимают 1-D и 2-D.
Палитра (отдельный пункт ROADMAP) естественно ложится сюда же.

## 3. План миграции (фазы, по нарастанию риска)

**Фаза 0 — тесты-сеть (обязательна перед рефактором).**
Закрыть тестами то, что трогаем: pivot-трансформации `time-breakdown`
(сборка `rows`/`dates`/`cellRatio`/нормализация по периодам), формат/масштаб
значений. Сейчас этого нет, а именно в pivot были последние баги.
Точки: рядом с `shared/ui/time-breakdown`, `shared/lib/utils` (масштаб).

**Фаза 1 — общий стиль-субстрат (низкий риск, чистый рефактор).**
Вынести в `shared`: палитру (источник цветов по ключу серии), themed-тултип,
хелпер формата/масштаба. Подключить в ОБА пути без смены поведения
(те же цвета по умолчанию). Гейт зелёный = поведение не изменилось.

**Фаза 2 — `ChartViewConfig` как контракт.**
Завести тип + протащить в `TimeBreakdownSection` те настройки, что уже есть в
1-D: палитра (из субстрата) и, где осмысленно, `chartStyle` серий. Закрывает
ROADMAP-пункт «2-D не уважает стиль графиков» и подключает палитру к 2-D.

**Фаза 3 — палитры (фича поверх контракта).**
UI выбора палитры (на группе/шаблоне — решить при реализации); запись в
`ChartViewConfig.paletteId`; оба пути читают палитру из общего источника.
Затем — конструктор своих палитр (отдельный пункт ROADMAP).

**Фаза 4 (north star, не сейчас) — новые типы чартов.**
Поверх единого контракта добавлять heatmap (ложится на 2-D-сетку
категория×время), treemap/sunburst (доли в иерархии, переиспользуют
`normalizeBy` и дерево), area. Каждый новый тип — выбор в `chartTypes`,
а не новый код-путь.

---

## 4. Риски и как их снимаем

- **Регрессии в pivot/нормализации.** → Фаза 0 (тесты) строго первой.
- **«Большой рефактор».** → Нарезано на фазы; Фаза 1 — чистый вынос без
  смены поведения, проверяется тем, что гейт и вид не меняются.
- **Соблазн объединить рендереры.** → Сознательно НЕ объединяем: оси
  различны по смыслу. Объединяем только настройки и стиль-примитивы.
- **Совместимость сохранённых конфигов.** → `ChartViewConfig` вводим с
  дефолтами (все поля опциональны), старые группы/дашборды читаются как есть.
