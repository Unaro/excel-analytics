# Агрегатные функции в формулах (Фаза A)

Статус: реализовано (Фаза A). Фаза B (удаление отдельного типа `aggregate`
+ миграция) — отдельным заходом.

## Зачем

Раньше агрегат для переменных calculated-формулы был **захардкожен `SUM`**
(query-compiler), то есть `a/b` всегда считалось `SUM(a)/SUM(b)`, а взять
MAX/AVG можно было только через отдельный «агрегатный» тип шаблона с одной
функцией. Теперь агрегат задаётся прямо в формуле: `MAX(a)/SUM(b)`,
`(MAX(a)/SUM(a)) - MIN(b)`.

## Как устроено (переиспользование пайплайна)

Ключевая идея — **препроцессор формулы** перед существующим движком,
а не переписывание движка.

`shared/lib/computation/lib/aggregate-formula.ts` (+ тесты):
`preprocessAggregateFormula(formula, fieldBindings, metricAliases, options)`
- находит вызовы `FN(переменная)` (SUM/AVG/MIN/MAX/COUNT/COUNT_DISTINCT/MEDIAN);
- регистрирует field-зависимость с этим агрегатом, ключуя по паре
  «колонка+агрегат» (`MAX(a)` и `SUM(a)` — разные зависимости `ca__MAX`, `ca__SUM`);
- переписывает формулу на пред-агрегированные псевдо-переменные:
  `MAX(a)/SUM(b)` → `ca__MAX / cb__SUM`;
- голую колонку вне агрегата авто-оборачивает в дефолтный агрегат
  (или, в строгом режиме, возвращает ошибку);
- ссылки на метрики не трогает (они уже агрегированы).

Дальше **всё работает как раньше**: base-CTE агрегирует
(`MAX(col) AS …__MAX`), `formula-to-sql` считает по псевдо-переменным,
`aggregation.ts` переагрегирует «Итого». Ядро (post-process, formula-to-sql)
не менялось.

### Ограничения Фазы A (валидация)

- агрегат применяется к ОДНОЙ переменной-колонке (`SUM(a+b)` — ошибка);
- без вложенных агрегатов (`SUM(MAX(a))` — ошибка);
- агрегат не над метрикой;
- скалярные функции (`round`, `abs`) допускают агрегаты внутри:
  `round(SUM(a)/SUM(b), 2)`.

### «Итого» (aggregation.ts)

Переагрегация field-зависимости теперь зависит от её агрегата:
SUM/COUNT → сумма, MIN → минимум, MAX → максимум, AVG → среднее по группам
(приближённо, без взвешивания), MEDIAN/COUNT_DISTINCT → null. Раньше всегда
суммировалось (для старого SUM-only — то же поведение).

## Настройки

`entities/app-settings` (persist) + секция «Агрегатные формулы» в настройках:
- **дефолтный агрегат** для голой колонки (по умолчанию SUM —
  сохраняет поведение старых формул);
- **строгий режим** — запрет голых колонок (нужен явный агрегат).

Прокидываются в `ClientComputeParams.formulaOptions` из хуков
(dashboard / group-view / kpi), входят в configHash → смена настройки
вызывает пересчёт. Дефолт (без params) — `{ SUM, requireExplicit:false }`,
поэтому существующие формулы и серверный путь работают без изменений.

## Конструктор формул

`VisualFormulaBuilder`: агрегатные функции вставляются структурным узлом
`FN(слот)` с выбором переменной в слоте (а не голым токеном `MAX`); round-trip
парсинг сворачивает `FN(var)` обратно в узел. Скалярные функции — обычные
токены. `extractVariables` отличает имена функций от переменных, поэтому
`requiredVariables` для привязки = только реальные колонки.

## Совместимость

Существующие calculated-шаблоны (голые переменные `a/b`) → авто-обёртка SUM →
идентичное прежнему поведение, без переписывания формул. Тип `aggregate`
продолжает работать как раньше (отдельный путь) — его удаление в Фазе B.

## Граница

Движок вычислений (worker/pg-engine) и формат ключей `${groupId}__${metricId}`
не трогались — только препроцессинг формулы и агрегат-зависимая
переагрегация «Итого».
