# Временна́я группировка breakdown (группировка по дате)

Дата: 2026-06-11.

## Что это

Если в настройках колонок датасета есть колонка с классификацией **«Дата»**
(`classification: 'date'`), на странице группы показателей появляется
переключатель измерения разбивки: вместо следующего уровня иерархии breakdown
строится по временны́м корзинам выбранной размерности —
**минуты / часы / дни / недели / месяцы / годы**. Таблица и графики при этом
показывают динамику метрик во времени; текущие фильтры пути продолжают
применяться в WHERE.

## Как устроено

### Компилятор (`query-compiler.ts`)

Новый параметр `ClientComputeParams.groupByDateGranularity: DateGranularity`.
Когда он задан, `groupByColumn` трактуется как дата-колонка:

- метка группы: DuckDB — `strftime(date_trunc('<g>', TRY_CAST(col AS TIMESTAMP)), fmt)`,
  PostgreSQL — `to_char(date_trunc('<g>', col::timestamp), fmt)`;
- `GROUP BY` — по этому же выражению;
- `ORDER BY "_group_label" ASC` — формат метки фиксирован на размерность
  (`%Y-%m-%d %H:%M` … `%Y`) так, что текстовая сортировка совпадает
  с хронологической;
- `LIMIT BREAKDOWN_LIMIT + 1` и флаг `breakdownTruncated` работают как
  для обычной группировки.

**Безопасность:** размерность интерполируется в SQL внутри `date_trunc('<g>',…)`,
поэтому значение строго проверяется по whitelist `DATE_GRANULARITIES`
(невалидное значение логируется и отбрасывается — обычная группировка),
а на PG-пути дополнительно валидируется Zod-enum в
`ClientComputeParamsSchema` до попадания в серверный компилятор.
Покрыто тестами (`query-compiler.test.ts`, включая инъекцию через размерность).

### Поведение типов данных

- DuckDB: `TRY_CAST` — значения, не приводимые к TIMESTAMP, дают NULL,
  такие строки отфильтровываются по пустой метке (как и раньше);
- PostgreSQL: `::timestamp` — колонка должна быть date/timestamp или
  ISO-текстом; ошибка приведения отображается как ошибка расчёта.

### UI (`widgets/group-view`)

- `use-group-breakdown`: состояние `dateGranularity`, первая колонка
  с классификацией «Дата» (`dateColumn`); в режиме дат `groupByColumn`
  подменяется на неё, `configHash` включает размерность (кэш и пересчёт
  работают штатно), drill-down отключён;
- `GroupViewContent`: селектор «По иерархии / По дате: …» (виден только
  при наличии дата-колонки) рядом с выбором типа графиков;
- `GroupBreakdownTable`: новый проп `dimensionLabel` — заголовок
  «Разбивка по: Дата · месяцы» при `nextLevel = null`.

## Ограничения и возможные продолжения

- Одна дата-колонка: используется первая с классификацией «Дата».
  Если потребуется выбор между несколькими — селектор колонки добавляется
  в тот же блок UI.
- Дашборды: основная таблица дашборда не показывает breakdown, поэтому
  переключатель добавлен на страницу группы. Перенос временно́го измерения
  в charts-секцию дашборда — отдельная задача (params уже всё поддерживают:
  достаточно передать `groupByColumn` + `groupByDateGranularity`
  из `use-dashboard-computation`).
- Размерность не персистится (сбрасывается при уходе со страницы) —
  осознанно, как runtime-состояние просмотра.
