# Дизайн: пользовательские типы со справочниками (ОКТМО/ОКАТО)

Статус: проект (зафиксировано 2026-06-12, реализация не начата).
Постановка владельца: см. ROADMAP.md → «Поддержка пользовательских типов».

## Цель

Пользователь загружает справочник (CSV/Excel до ~50 МБ, например ОКТМО или
ОКАТО), настраивает его — и получает **новый тип колонки** в настройке
датасета. Колонка с кодами (`01512000106`) после назначения типа всюду
отображается названием территории. В перспективе из справочника
подтягиваются и другие колонки (через JOIN к датасету).

## Инженерные факты, определяющие дизайн

1. **Конвейер импорта уже есть.** Worker `IMPORT_EXCEL` принимает CSV/xlsx,
   батчит по 25k строк, показывает прогресс, персистит Arrow в IndexedDB.
   Справочник можно везти этим же конвейером — второй импорт не нужен.
2. **Excel ест ведущие нули.** Код `01512000106` приходит числом
   `1512000106`. Поэтому «правило подготовки ключа» — обязательная часть
   типа: хранить как текст + `LPAD(CAST(col AS VARCHAR), N, '0')`,
   где N = 8 или 11 (ОКТМО/ОКАТО имеют переменную длину).
3. **PG-датасеты не видят браузерный DuckDB.** SQL для PG исполняется на
   сервере пользователя — JOIN со справочником, живущим в браузере,
   там невозможен. Для PG нужна клиентская подстановка (или выгрузка
   справочника в PG — этап 3).

## Варианты

### A. Справочник = служебный датасет + JOIN в SQL (целевой)

- Справочник — датасет с ролью `reference` (скрыт из переключателя).
- Сущность «пользовательский тип»:
  `{ id, name, dictionaryDatasetId, keyColumn, displayColumn,
     keyNormalization: 'none' | 'lpad8' | 'lpad11' }`.
- `ColumnConfig.customTypeId?: string`.
- Компилятор: `LEFT JOIN ref ON <normalize>(t.col) = ref.key`,
  метка — `COALESCE(ref.display, t.col)`.

✅ Поиск/фильтры/сортировка по названиям; доступ к остальным колонкам
справочника. ❌ Дороже всего; не работает для PG напрямую.

### B. Клиентский словарь подстановки (быстрый, без SQL)

`Map<код, название>` в IndexedDB, ленивая загрузка; подстановка меток
`_group_label` после вычисления (post-processing на клиенте).

✅ Быстро; одинаково для DuckDB и PG; ноль правок SQL.
❌ Поиск/фильтры/иерархия остаются по кодам; другие колонки справочника
недоступны; весь ОКТМО (~280k строк) в памяти вкладки.

### C. Материализация при импорте

Разовый `CREATE TABLE AS … JOIN` добавляет колонку `<col>_название`
в таблицу датасета.

✅ Ноль рантайм-сложности после. ❌ Обновление справочника = переимпорт
всех датасетов; для PG невозможно; дубль данных. **Отклонён** как основной
путь (годится разве что как оптимизация этапа 2).

## Решение: A как цель, поэтапно, с механикой B для PG

### Этап 1 — фундамент + видимый результат

1. Роль `reference` у датасета (поле в dataset-store + скрытие в UI).
2. `entities/reference-type` — persist-стор типов (id, name, справочник,
   ключ, отображение, нормализация).
3. `features/manage-references` — страница «Справочники»: загрузка файла
   (существующий конвейер), выбор ключевой/отображаемой колонки,
   нормализация, предпросмотр матчинга («N кодов из примера найдены»).
4. `ColumnConfig.customTypeId` + пункт в настройке колонки
   (рядом с classification).
5. Подстановка названий на клиенте (механика B): хелпер
   `resolveReferenceLabels(items, dict)` в post-processing breakdown.

Результат: «вместо кода — территория» работает на file и PG.

### Этап 2 — JOIN в DuckDB-компиляторе

- `ClientComputeParams.referenceJoins?: Array<{ column, refTable,
  refKey, refDisplay, normalization }>` (строится хуками из column-config).
- query-compiler: LEFT JOIN + COALESCE-метка; whitelist нормализаций
  (как с date granularity — значения интерполируются в SQL).
- Поиск/сортировка/фильтры по названиям; доп. колонки справочника
  как поля датасета.

### Этап 3 — PG (по потребности)

Либо остаёмся на клиентской подстановке, либо выгрузка справочника
во временную таблицу PG-базы пользователя (требует прав на запись —
обсудить отдельно).

## Раскладка по FSD

| Что | Где |
|---|---|
| Роль датасета `reference` | `entities/dataset/model/store.ts` (+ миграция persist) |
| Стор пользовательских типов | `entities/reference-type/` (новый слайс) |
| UI управления справочниками | `features/manage-references/` + страница в `app/` |
| `customTypeId` колонки | `shared/lib/types/dataset.ts` (ColumnConfig) + UI setup-wizard |
| Подстановка меток (этап 1) | `shared/lib/computation/lib/` (хелпер) + хуки страниц |
| JOIN (этап 2) | `query-compiler.ts` + `ClientComputeParams` + Zod-схема сервера |

## Открытые вопросы

- Фильтрация по названиям нужна с первого дня? (Если да — этап 2
  поднимается в приоритете, этап 1 сжимается.)
- Обновление справочника новой версией файла: полная замена таблицы
  (проще) или версионирование?
- Несколько отображаемых колонок (название + уровень + родитель) —
  отложено до этапа 2.
