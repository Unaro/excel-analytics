# Единый импорт: один настраиваемый путь для сырых данных и агрегатов

Дизайн-документ. Статус: **черновик на утверждение** (код не трогали).
Дата: 2026-06-24. Цель — свести два пути импорта (сырые таблицы и
файлы-агрегаты) в один настраиваемый, чтобы пользователь при загрузке сразу
задавал структуру и показатели, а «сырой» путь постепенно вышел из обихода.

---

## 1. Что есть сейчас (факты из кода)

Импорт — **один визард** ([use-setup-wizard.ts](../../src/widgets/setup-wizard/model/use-setup-wizard.ts)):
шаги `manager → upload → import → columns`. Тоггл `isAggregate` переключает
последний шаг между двумя панелями:

### Сырой путь (`isAggregate = false`)
[ColumnSetupStep](../../src/widgets/setup-wizard/ui/ColumnSetupStep.tsx):
классификация колонок (ключ/мера/дата/…), типы. На выходе — **датасет**.
**Группы показателей НЕ создаются** — пользователь идёт в `/groups/new`
(конструктор) и собирает их вручную отдельным действием.

### Агрегатный путь (`isAggregate = true`)
[AggregateStructurePanel](../../src/widgets/setup-wizard/ui/AggregateStructurePanel.tsx)
→ `AggregateLayoutConfig` → [sync-engine.ts](../../src/features/setup-dataset/model/sync-engine.ts).
Делает СРАЗУ много:
- разметка иерархии по **каскаду ключевых колонок** (уровни) +
  «шапка»-группировка колонок-мер;
- сплющивание листьев в CSV + узлы (`setNodes`) с «введёнными» значениями;
- `addLevel` — уровни иерархии датасета;
- `createAggregateGroups` — **создаёт шаблоны метрик, группы и метрики**:
  на каждый логический показатель (имя) — шаблон (формула/формат/`normalizeBy`),
  на каждую колонку-меру — метрика с привязкой `fieldAlias → columnName`.

### Ключевое наблюдение
**Агрегатный путь — это надмножество сырого.** Всё, что делает сырой
(датасет + классификация колонок), агрегат тоже делает, и СВЕРХУ добавляет:
иерархию по каскаду ключей, узлы с введёнными значениями, и **создание
групп/шаблонов/метрик прямо при загрузке**.

То есть «сырой = агрегат без каскада ключей» (плоские данные, один уровень,
без узлов). А недостающая у сырого ценность — это именно **создание групп
при импорте**, которое у агрегата уже есть в `createAggregateGroups`.

---

## 2. Что общего, что различается

**Общее (есть/нужно обоим):**
- классификация колонок и типы (`ImportParams`);
- шаблоны показателей: формула + формат + единица + `normalizeBy`
  (блок «Формула и формат» в агрегатной панели — Фаза 1);
- создание групп и метрик (`createAggregateGroups` — пока только агрегат);
- привязка `fieldAlias → columnName`.

**Специфично для агрегата:**
- каскад ключевых колонок → уровни иерархии;
- узлы (`setNodes`) с «введёнными» значениями уровней;
- сплющивание листьев.

Вывод: объединять нужно **панель разметки**, а «агрегатность» свести к
**опции** «данные иерархические (каскад ключей)» внутри единой панели. Без
каскада — плоский датасет (бывший «сырой»), с созданием групп по желанию.

---

## 3. Оставшиеся фазы

### Фаза 1b — многополевые формулы в импорте
Сейчас `analyzeFormula` в панели **запрещает** >1 поля. Движок их уже умеет
(`preprocessAggregateFormula` + `query-compiler`, см.
[aggregate-formulas.md](aggregate-formulas.md)), и конструктор групп —
тоже (`use-group-builder`). Нужен лишь **UI-режим «расчётный показатель»** в
`AggregateStructurePanel`: формула вида `SUM(a)/SUM(b)`, привязка КАЖДОГО
алиаса к своей колонке (a→колонка, b→колонка) → **одна** метрика (а не
«колонка = метрика»). Сериализация — расширить `AggregateTemplateSpec`
(несколько `fieldBindings` на спеку) и ветку в `createAggregateGroups`
(сейчас один `fieldBindings[0]`).

### Фаза 3 — создание групп для сырых данных
Дать в сыром пути ту же возможность «создать показатели и группы при
загрузке», что у агрегата. Технически — переиспользовать
`createAggregateGroups` для плоских данных (без каскада ключей: один
«уровень», без узлов). Это убирает главное различие ценности между путями.

### Фаза 4 — слияние панелей
Свести `ColumnSetupStep` и `AggregateStructurePanel` в **одну панель** с
опцией «иерархические данные (каскад ключей)». «Сырой» режим = опция выключена.
Тоггл `isAggregate` уходит; остаётся одна настраиваемая разметка. Старый
отдельный путь — в депрекейт.

---

## 4. План миграции (фазы, по нарастанию риска)

**Фаза 0 — тесты-сеть.** Покрыть `createAggregateGroups` (сборка
шаблонов/групп/метрик, дедуп шаблона по имени+формуле, привязки) и сериализацию
`metricTemplateSpecs` round-trip — это ядро, которое будут трогать все фазы.
Сейчас прямых тестов на него нет.

**Фаза 1b — многополевые формулы** (контейнерно, низкий риск): режим
«расчётный показатель» в панели + мульти-`fieldBindings` в спеке и
`createAggregateGroups`. Движок не трогаем.

**Фаза 3 — группы для сырых** (средний риск): опция «создать группы при
загрузке» в сыром шаге; переиспользовать `createAggregateGroups` для плоских
колонок. UI-сближение с агрегатной панелью.

**Фаза 4 — слияние** (крупный, делать последним): единая панель, опция
«иерархия (каскад ключей)», депрекейт отдельного «сырого». Сохранить обратную
совместимость сохранённых `AggregateLayoutConfig` и сырых датасетов.

---

## 5. Риски

- **Регрессии в импорте — самом критичном входе данных.** → Фаза 0 (тесты)
  строго первой; каждая фаза проходит полный гейт.
- **Слом модели «колонка = метрика»** при многополевых формулах. → 1b — это
  отдельный РЕЖИМ спеки (`kind: 'perColumn' | 'calculated'`), обычные шаблоны
  не меняются.
- **Совместимость конфигов.** → `AggregateLayoutConfig` расширяем только
  опциональными полями; старые конфиги читаются как есть (round-trip тест).
- **Большой объём Фазы 4.** → нарезано; 1b и 3 самостоятельно полезны и не
  требуют 4.

## 7. Рекомендация

Порядок: **Фаза 0 (тесты) → 1b → 3 → 4**. 1b завершает историю формул в
импорте малой кровью (движок готов), 3 убирает главное различие ценности
(группы для сырых), 4 — финальное слияние. Каждая фаза самостоятельно полезна.
