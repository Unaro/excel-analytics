# Дизайн: парсинг файлов-агрегатов (мега-босс)

Зафиксировано 2026-06-21. Это план, не реализация. Задача из
[../ROADMAP.md](../ROADMAP.md) («🦖 Мега-босс»). Делать строго после
производительности и оптимизаций — они закрыты.

## Цель

Принимать НЕ только плоские таблицы сырых строк, но и уже
агрегированные/иерархические файлы — как их строит Excel:

1. **Сгруппированная «шапка»** — над столбцами есть объединяющий заголовок
   верхнего уровня (multi-row header, объединённые ячейки).
2. **Группировка строк по уровням** (excel-outline): строка несёт уровень
   иерархии, а не только листовые значения.
3. **Разный набор колонок на разных уровнях** — колонка осмысленна только на
   части уровней (напр. «Площадь» есть у районов, у дворов пусто). Схема не
   плоская.
4. **Предпосчитанные/ручные метрики на уровнях** — на узле может стоять готовая
   величина (посчитана заранее или вписана руками), которая НЕ сходится с
   суммой детей. Её нельзя молча пересчитывать: храним как отдельную переменную
   и даём сравнить «введённое vs вычисленное» (расхождение — сама по себе
   аналитическая ценность).

## Структура подопытного файла (реальный пример)

Скрин предоставлен владельцем (территории Тюмени, функциональные зоны → ДОО).
Разбор:

- **Multi-row шапка (#1):** строка 1 — групповые спаны (`Расположение`,
  `Параметры функциональной зоны`, `Дошкольные образовательные организации`),
  строка 2 — реальные имена колонок. Составное имя = «группа · колонка».
- **Уровень кодируется КАСКАДОМ ключевых колонок**, а не Excel-outline:
  `Наименование1 → Наименование2 → Наименование3` (гео) → `Планировочный` (`8`)
  → `Идентификатор` (`8:01:06` — зона) → `GUID` (`{78b0d545…}` — лист-объект).
  **Уровень строки = индекс самого правого заполненного ключа.**
  **Лист = заполнен самый глубокий ключ (GUID).**
- **Зелёные строки = узлы-агрегаты** (зона `8:01:06`, `8:02:04`) с
  предпосчитанными метриками (`0,144 / -0,144`, `Потребность/Мощность/Оценка`) —
  это «введённые» значения (#4). Заливка — подтверждающий сигнал (не основной).
- **«Итого» сверху:** строки `город Тюмень` (только `Наименование`, без кодов)
  несут городские суммы (`Численность 1639132`, `Плотность 33,65` …) — это
  верхнеуровневые узлы. Явная строка `Итого/Всего` тоже возможна (см. ниже).
- **Разреженность (#3):** `Площадь/Статус ФЗ` — у листьев; `Потребность/Мощность`
  — у зон; `Численность/Плотность` — у города. На «чужом» уровне — пусто/0.
- **Коды `8`, `8:01:06`** — это категориальные ключи (ведущие нули, двоеточия),
  НЕ числа/время. Импорт уже умеет держать их строкой (шаг «Импорт», VARCHAR —
  фаза 3b setup'а). На каскад ключей это распространяется напрямую.

**Вывод для дизайна:** в этом классе файлов уровень читается из ЗНАЧЕНИЙ
(каскад ключей), без `!rows`/стилей. Это упрощает детект и убирает зависимость
от `cellStyles` для ядра — заливка остаётся опциональным подтверждением.

## Пустота и нули — настройка на импорте

**Решение (от владельца):** `0` — это настоящий ноль, а не пропуск. Пустые
ячейки тоже встречаются и означают «нет значения» (разреженность #3). Поскольку
в разных файлах граница «что считать пустотой» гуляет, выносим её в ЯВНУЮ
настройку на шаге парсинга:

- по умолчанию пусто = только реально пустая ячейка (`null`/`""`); `0` —
  значение;
- пользователь может добавить трактовку пустоты: `0`, прочерк `—`, `н/д`, `-`
  и свои сентинелы;
- настройка влияет на: разреженность (применима ли колонка к уровню),
  агрегаты (пустое не участвует в SUM/AVG, ноль — участвует) и Δ
  «введённое vs вычисленное».

*Точка входа:* тот же шаг «Импорт/Структура» (`widgets/setup-wizard`,
`features/setup-dataset/lib/file-preview.ts`), рядом с десятичным разделителем и
типами колонок.

## Строки «Итого»/«Всего» и узлы-агрегаты

Ключевое наблюдение: правило **«лист = самый глубокий заполненный ключ»** само по
себе отделяет агрегаты от листьев. Всё, что не лист (город, планировочный район,
зона, явная строка «Итого»), — это УЗЕЛ. Поэтому:

- **Фаза 1 (сплющивание листьев) берёт ТОЛЬКО листовые строки** → «Итого» и
  зелёные узлы автоматически не попадают в роллап, двойного счёта нет.
- **Фаза 2 поднимает узлы**: их метрики — это `enteredValue` соответствующего
  узла, сравниваем с вычисленным по детям (`discrepancy`).
- **Явная строка `Итого/Всего/Total/Итоговая`** (может стоять в начале или
  конце, иногда без каскада ключей — просто метка в первой колонке) детектится
  по ключевым словам + позиции. Трактуем как узел корня (или родителя секции),
  НЕ как лист. Никогда не агрегируем её как данные.

## Инженерные факты, определяющие дизайн

- **Движок считает плоским GROUP BY.** `query-compiler` строит SQL-агрегаты по
  колонкам (`groupByColumn`/даты). Иерархия сейчас выводится из колонок: уровень
  = колонка (`entities/hierarchy`), фильтр пути = WHERE по колонкам-предкам.
  Роллап «снизу вверх» — это GROUP BY по колонкам-предкам, и он работает ТОЛЬКО
  если в каждой листовой строке проставлены метки всех предков.
- **Парсер сейчас читает ОДНУ строку заголовка** (`excel-parser.ts:extractHeaderRow`,
  range-limited) и материализует плоские `flatRows` (`parseExcelToJson`). Ни
  объединённых шапок, ни уровней строк он не знает.
- **SheetJS даёт хуки для структуры**, но не по умолчанию:
  - объединённые ячейки шапки — `ws['!merges']` (всегда доступны);
  - outline-уровень строки — `ws['!rows'][r].level` (если есть в файле);
  - отступ-индент ячейки — `cell.s.alignment.indent` (нужен `cellStyles:true`
    при чтении; иначе уровень определяем эвристикой по пустым ведущим колонкам).
- **value / displayValue уже разделены** в модели (см.
  [reference-types.md](reference-types.md)) — метку узла показываем, ключ
  храним. Это пригодится для меток уровней.
- **Результат вычисления плоский** (`ComputedMetricValue.value`): места под
  «введённое значение» и «расхождение» нет — добавим (вызов #4).

## Варианты представления иерархии

### A. Сплющить в листья на импорте (быстро, но теряет суть)
Развернуть агрегат в листовые строки: метки предков протащить вниз в
колонки-предки, предпосчитанные строки-узлы выбросить. **Плюс:** движок не
меняется вообще. **Минус:** теряются #3 (колонки уровня) и #4 (введённые
значения узлов) — а #4 и есть главная ценность. Как самостоятельное решение
не годится, но это валидная ПЕРВАЯ фаза (см. ниже).

### B. Явная модель узлов (целевая)
Импорт строит таблицу узлов: строка = узел `(nodeId, parentId, level, label,
колонки…)` + блок `enteredValues` для предпосчитанных метрик. Движок агрегирует
снизу вверх от листьев и ОТДЕЛЬНО отдаёт сохранённое введённое значение узла для
сравнения. **Плюс:** закрывает все 4 пункта. **Минус:** крупное изменение —
модель датасета, парсер, и путь вычислений (узлы + entered рядом с computed).

**Решение:** идём к B, но поэтапно — фаза 1 на базе A даёт пользователю быстро
импортировать реальные файлы, фаза 2 добавляет узлы и «введённое vs
вычисленное», фаза 3 — полную разреженную схему.

## Фазы

### Фаза 0 — Детект структуры + шаг подтверждения (фундамент)
Расширить превью импорта: распознать multi-row шапку (`!merges`) и **каскад
ключевых колонок** (упорядоченный список колонок-меток: Наименование1..3,
Планировочный, Идентификатор, GUID). Из каскада выводим:
- **уровень строки** = индекс самого правого заполненного ключа;
- **лист** = заполнен самый глубокий ключ; **узел** = иначе;
- **строки «Итого/Всего/Total»** = по ключевым словам в колонке-метке + позиция.

Показать под-шаг «Структура» в визарде, где пользователь подтверждает/правит:
порядок ключевых колонок (каскад), сколько строк шапки, какие колонки — метрики,
трактовку пустоты (см. выше). Заливка (зелёные узлы) — опциональное подтверждение
через `cellStyles`.

**Внизу шага — предпросмотр результата (обязательно):**
- **Какие группы создадутся** — список будущих групп показателей (по составным
  именам из шапки, напр. `ДОО · Потребность`, `Параметры зоны · Плотность`) с
  чекбоксами: пользователь выбирает, что создавать, а что нет.
- **Предварительная иерархия** — дерево из каскада ключей (город → планировочный
  → зона → объект) на сэмпле, чтобы видеть, что уровни распознаны верно.

Движок НЕ трогаем — только парсинг + модель разметки (`AggregateLayout`) +
предпросмотр.
*Точки входа:* `excel-parser.ts` (чтение `!merges`, опц. `cellStyles`),
`features/setup-dataset/lib/file-preview.ts` (детект каскада/итого/пустоты),
`widgets/setup-wizard` (новый шаг + предпросмотр групп и дерева),
`AggregateLayout` в `entities/dataset`.

### Фаза 1 — Сплющивание листьев (MVP, движок без изменений)
По подтверждённой разметке материализовать плоскую таблицу ЛИСТЬЕВ: метки
предков протащить в колонки-предки каждого листа, импорт — текущим путём
(`importExcelBuffer`). Иерархия строится из колонок-предков как сейчас.
Предпосчитанные значения узлов пока НЕ сохраняются (задокументированное
ограничение). Результат: пользователь импортирует реальные `.xlsx`-агрегаты и
работает с ними для обычного случая «сумма листьев».
*Точки входа:* `duckdb/worker.ts` (сборка leaf-таблицы из разметки),
`features/setup-dataset/model/sync-engine.ts`.

### Фаза 2 — Таблица узлов + «введённое vs вычисленное» (ядро ценности)
Сохранять полную структуру узлов (не только листья): параллельное node-
представление `(nodeId, parentId, level, label, enteredValues)`. Движок
по-прежнему агрегирует листья, но дополнительно отдаёт `enteredValue` узла и
`discrepancy = entered − computed`. UI группы/дашборда показывает
введённое/вычисленное/Δ.
*Точки входа:* `entities/dataset/model/types.ts` (node-схема, флаг `isAggregate`),
`shared/lib/types/computation.ts` (`enteredValue`/`discrepancy` в
`ComputedMetricValue`/`VirtualMetricValue`), движок
(`shared/lib/computation`), `widgets/group-view` (колонки Δ).

### Фаза 3 — Разреженная схема и шапки (полировка)
Полная поддержка «колонка только на части уровней», составные имена из
multi-row шапки, привязка метрик к конкретным уровням. Доводка UI разметки.
*Точки входа:* `entities/dataset` (схема по уровням), `entities/hierarchy`
(уровни из outline, а не только из колонок), `query-compiler` (учёт
применимости колонки к уровню).

## Открытые вопросы (решить до фазы 2)

1. ~~Детект листа без outline.~~ **Решено реальным файлом:** уровень = глубина
   каскада ключевых колонок (значения), `!rows`/стили не нужны для ядра. Остаётся
   ручное подтверждение каскада на шаге «Структура» (фаза 0).
2. ~~Привязка по колонке или шапке.~~ **Решено:** по составному имени из шапки
   (`группа · колонка`, напр. `ДОО · Потребность`). Внизу шага «Структура»
   пользователь выбирает чекбоксами, какие группы создавать.
3. **Несколько листов** в одном файле — объединять, выбирать, игнорировать?
4. **Поведение Δ при многоуровневой агрегации** — сравнивать введённое узла
   только с прямыми детьми или со всем поддеревом листьев?
5. **Промежуточные узлы без явного «Итого»** (зелёные строки) vs явная строка
   `Итого` — это один механизм (узел) или их надо различать в UI?
6. ~~0 vs пусто.~~ **Решено:** `0` — настоящий ноль; пустота настраивается
   пользователем на импорте (по умолчанию пусто = только пустая ячейка). См.
   раздел «Пустота и нули».

## Рекомендация по порядку

Фаза 0 → Фаза 1 (быстрый видимый результат: файлы импортируются) → Фаза 2
(уникальная ценность: введённое vs вычисленное) → Фаза 3 (полнота). Каждая фаза
проходит гейт (type-check + lint + test + build) и остаётся пригодной к
использованию.
